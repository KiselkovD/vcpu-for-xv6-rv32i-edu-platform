# vcpu-xv6-rv32i-educational-platform

Данный проект представляет собой образовательный комплекс для изучения операционной системы **xv6** (портированной на 32‑битную RISC‑V) и разработанного в рамках магистерской диссертации **токенного процессора `vcpu`** – многотактового ядра с явной синхронизацией стадий. Комплекс позволяет:

- Собирать и запускать `xv6-rv32` в QEMU.
- Собирать и симулировать `vcpu` (Verilog) с загруженным образом `xv6` или пользовательской программой.
- Сравнивать трассы выполнения `vcpu` с эталонной моделью QEMU.

## Структура проекта

```
.
├── Dockerfile                # контейнер со всеми инструментами
├── run.sh                    # скрипт запуска Docker-контейнера
├── docs/                     # теоретические материалы (концепции, введения)
├── xv6-rv32/                 # исходный код ОС xv6 для RISC‑V 32
├── vcpu/                     # исходный код Verilog процессора vcpu
└── qqq/                      # служебная директория (не используется)
```

## Порядок изучения

Рекомендуется двигаться от простого к сложному, сначала разобравшись с теорией и средой, затем – с ОС, и наконец – с аппаратной реализацией.

### 1. Теоретическая подготовка
- Ознакомиться с концепциями RISC‑V (регистры, система команд, привилегированные режимы).
- Прочитать файлы в папке `docs/`:
  - `Введение_vcpu.md`, `Концепции_vcpu.md` – принципы токенной синхронизации.
  - `Введение_xv6.md`, `Концепции_xv6.md` – архитектура ядра xv6, планировщик, память.

### 2. Работа с операционной системой xv6
- **Сборка и запуск в QEMU** – см. раздел «Сборка и запуск xv6».
- Изучение кода ядра по рекомендуемой последовательности (см. `xv6-rv32/README.md` или документацию).
- Отладка с GDB, изучение системных вызовов, файловой системы.

### 3. Изучение процессора vcpu
- **Сборка и симуляция** пользовательских программ (asm → hex → vvp).
- **Запуск xv6 на vcpu** (требуется исправление несостыковок, см. раздел «Важные замечания»).
- Анализ трассировки и сравнение с QEMU.

### 4. Сравнительная верификация
- Использование скрипта `compare_traces.py` для поиска расхождений между `vcpu` и эталоном.
- Эксперименты с отключением/включением трассировки.

## Сборка и запуск xv6

Все действия выполняются внутри Docker-контейнера (см. раздел «Docker»). Вне контейнера также можно работать, если установлены: `riscv32-unknown-elf-gcc`, `qemu-system-riscv32`, `make`, `perl`, `gdb`.

```bash
# Перейти в директорию xv6
cd xv6-rv32

# Собрать ядро и пользовательские программы
make

# Запустить в QEMU (обычный режим)
make qemu

# Запустить с отладкой (GDB на порту 25000)
make qemu-gdb
```

Для выхода из QEMU: `Ctrl+A` затем `X`.

**Очистка:** `make clean` (удаляет `.output/` и сгенерированные файлы).

## Сборка и запуск vcpu

### Предварительные требования
- Icarus Verilog (≥ 10)
- Python 3 (для ассемблера `encode.py`)
- Perl (для преобразования порядка байт)
- (опционально) RISC‑V тулчейн для конвертации ELF → hex

Все эти зависимости присутствуют в Docker-образе.

### Работа с пользовательской программой

1. Поместите ассемблерный код в `vcpu/to_execute/prog.txt`.
2. Соберите симулятор и запустите программу:
   ```bash
   cd vcpu
   make build          # компиляция Verilog → testbench.vvp
   make run_prog       # ассемблирование, создание образа, симуляция
   ```
3. Результаты симуляции появятся в `vcpu/.output/`:
   - `trace.log` – основной лог,
   - `cpu_trace.log` – трассировка стадий токена,
   - `uart_output.txt` – вывод программы через UART.

### Запуск xv6 на vcpu (экспериментально)

**Важно:** текущая версия `vcpu` имеет несколько архитектурных несоответствий с `xv6` (endianness, объём памяти, обработка прерываний). Для полноценного запуска требуются доработки, описанные в разделе «Известные проблемы».

Тем не менее, запуск возможен по следующей схеме:

1. Собрать `xv6` и получить сырой образ `kernel.raw.hex`:
   ```bash
   cd xv6-rv32
   make
   make raw           # генерирует .output/kernel.raw.hex (little-endian)
   ```
2. Скопировать образ в директорию `vcpu/to_execute/`:
   ```bash
   cp .output/kernel.raw.hex ../vcpu/to_execute/
   ```
3. В директории `vcpu` выполнить:
   ```bash
   make run_xv6
   ```
   Процессор симулирует загрузку ядра, но, скорее всего, завершится ошибкой из-за нехватки памяти или неправильной работы прерываний.

### Сравнение с QEMU

Предварительно необходимо сгенерировать логи QEMU при выполнении той же программы (например, `prog.txt`). Для этого в `xv6-rv32` можно запустить QEMU с опциями логирования:

```bash
make qemu-debug      # (если такая цель определена) или вручную:
qemu-system-riscv32 -machine virt -bios none -kernel kernel/kernel -nographic -d in_asm,nochain -D qemu.log
```

Затем скопировать `qemu.log` и `qemu_execution.log` в `vcpu/qemu_logs/`. После этого в `vcpu/`:

```bash
make compare
```

Скрипт `scripts/compare_traces.py` сформирует отчёт `.output/compare_report.txt`.

## Docker

Для воспроизводимой среды предоставлен `Dockerfile` (см. ниже). Он включает:

- Базовый Ubuntu 22.04
- RISC‑V тулчейн `riscv32-elf` (с ссылками `riscv32-unknown-elf-*`)
- QEMU 5.0 для `riscv32-softmmu`
- Icarus Verilog, Python, Perl, make, gdb, hexdump, tree
- Настроенное автодополнение bash для `make`

### Сборка и запуск контейнера

```bash
# Из корня проекта
docker build -t riscv-xv6-env .
./run.sh
```

Скрипт `run.sh` монтирует текущую директорию в `/workspace` контейнера и запускает интерактивную оболочку bash. Внутри контейнера доступны все команды для сборки `xv6` и `vcpu`.

**Примечание:** `run.sh` уже присутствует в проекте; убедитесь, что он имеет право на выполнение (`chmod +x run.sh`).

## Известные проблемы и несостыковки

При интеграции `xv6` и `vcpu` выявлены следующие расхождения (подробнее – в отчёте):

1. **Порядок байт** – `vcpu` ожидает big-endian, а `xv6` (и RISC‑V) использует little-endian. Преобразование только для загрузки слов недостаточно.
2. **Объём памяти** – `vcpu` предоставляет 2 МБ ОЗУ, тогда как `xv6` с файловой системой требует десятки МБ.
3. **Игнорирование MIE/SIE** – `vcpu` не проверяет глобальные биты разрешения прерываний, что нарушает синхронизацию ядра.
4. **Таймерные прерывания** – в `vcpu` они подаются как SSIP, а не как machine timer interrupt.
5. **Упрощённый PLIC** – поддерживается только один источник (UART), не хватает интерфейса claim/complete.
6. **Отсутствие проверки прав доступа к CSR** – пользовательский режим может изменять системные регистры.

Эти проблемы делают невозможным стабильный запуск `xv6` на `vcpu` без доработок. Тем не менее, комплекс полезен для изучения отдельных подсистем и отладки.
